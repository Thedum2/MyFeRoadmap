# CSS는 어떤 문제를 해결하기 위해 등장하였는가?

#질문

초기 웹에서 HTML은 문서 구조를 표현하는 데는 훌륭했지만, 화면을 "어떻게 보이게 할지"를 통제하는 능력이 약했다. 같은 문서라도 글꼴, 여백, 정렬, 색상 같은 표현을 일관되게 적용하기 어려웠고, 페이지마다 반복 수정이 필요했다.

이 문제를 해결하기 위해 [[CSS]]가 등장한다. 핵심 선택은 단순하다. 문서 구조(HTML)와 표현 규칙(CSS)을 분리한다. 비유하면 건물의 골조와 인테리어 도면을 따로 관리하는 방식이다. 골조가 같아도 인테리어 규칙만 바꿔 전체 느낌을 바꿀 수 있다.

![[assets/images/CSS-박스모델.svg]]

low-level에서는 브라우저가 HTML로 DOM을 만들고, CSS를 파싱해 [[CSSOM]]을 만든다. 이후 DOM과 CSSOM을 결합해 렌더 트리를 생성하고 레이아웃 계산을 수행한 뒤 픽셀로 그린다. 이 파이프라인 덕분에 구조와 표현을 분리하면서도 최종 화면은 일관되게 계산된다.

CSS가 없던 시절에는 표현 변경이 문서 내용 수정과 섞여 유지보수가 어려웠다. CSS 도입 후에는 스타일 규칙을 재사용하고, 테마를 교체하고, 반응형 규칙을 선언하는 방식이 가능해졌다. 즉 "변경 비용"이 크게 줄었다.

물론 CSS도 한계가 있다. 상속, 우선순위, 특이성(specificity) 충돌 때문에 규모가 커지면 관리가 복잡해진다. 그래서 방법론(BEM), 전처리기, CSS-in-JS, 디자인 토큰 같은 운영 기법이 발전했다. 그럼에도 본질은 변하지 않는다. CSS는 HTML이 해결하지 못한 "표현 계층의 독립성" 문제를 해결하기 위해 태어났다.

## 🔎 확장 질문

★★★★★ CSS의 cascade와 specificity는 왜 강력하면서도 위험한가?

> [!important]
> 적은 코드로 전역 스타일을 제어할 수 있어 강력하지만, 우선순위 규칙이 복잡해지면 의도치 않은 덮어쓰기와 회귀 버그가 발생하기 쉽다. 규모가 커질수록 규칙 설계가 중요해진다.

★★★★☆ 반응형 디자인에서 CSS 미디어 쿼리는 어떤 구조적 이점을 주는가?

> [!important]
> 동일한 HTML 구조를 유지한 채 화면 조건별 표현만 분기할 수 있다. 콘텐츠와 표현을 분리했기 때문에 장치 다양성 대응이 선언적으로 가능해진다.

★★★☆☆ CSSOM과 렌더링 성능은 어떤 관계가 있는가?

> [!important]
> 스타일 변경은 CSSOM 재계산과 레이아웃/페인트 비용을 유발할 수 있다. 어떤 속성을 언제 바꾸는지에 따라 렌더링 비용이 달라지므로 성능 최적화는 파이프라인 이해를 전제로 한다.

## 🧠 이해 점검 퀴즈

**Q1 (단답형)** 브라우저가 CSS를 파싱해 만드는 객체 모델 이름은?

> [!important]
> CSSOM.

**Q2 (서술형)** CSS가 HTML의 어떤 한계를 해결했는지 설명하라.

> [!important]
> HTML에 섞여 있던 표현 책임을 분리해, 구조는 HTML이 담당하고 스타일은 CSS가 담당하게 만들었다. 이로써 일관된 스타일 적용, 재사용, 대규모 유지보수가 가능해졌다.

**Q3 (설계 의도)** 왜 스타일을 문서마다 직접 쓰지 않고 별도 스타일 언어로 분리했는가?

> [!important]
> 표현은 반복 변경이 잦고 매체별 요구가 다르다. 구조와 분리해야 변경 범위를 줄이고 재사용성을 높이며, 브라우저가 렌더링 파이프라인을 체계적으로 최적화할 수 있다.
