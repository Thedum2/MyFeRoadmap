# MPA vs SPA는 어떻게 다른가?
#질문

## 1. 등장 배경
- 초기 웹은 [[MPA]] 중심이었다.
- 페이지 이동마다 서버가 새 HTML을 생성했다.
- 구조적 한계는 "이동마다 전체 문서를 다시 내려받는다"는 점이었다.
- 화면 일부만 바뀌어도 CSS, JS 초기화와 렌더 트리가 다시 만들어졌다.
- 상호작용이 많아질수록 지연과 깜빡임이 커졌다.

## 2. 구조적 전환
- [[SPA]]는 "문서 1회 로드 + 데이터 갱신"으로 전환했다.
- 핵심 아이디어는 화면을 페이지 단위가 아니라 컴포넌트 단위로 갱신하는 것이다.
- 상태를 브라우저 메모리에 유지하고 API 응답으로 UI를 갱신한다.
- 트레이드오프는 초기 JS 번들 비용 증가, SEO/초기 표시 최적화 난이도 상승이다.

> 핵심: MPA는 네비게이션마다 문서 교체, SPA는 문서 유지 + 상태 기반 부분 갱신.

## 3. 내부 동작 원리 (Low-Level)
- 계층 구조
  - MPA: Browser -> HTTP Request -> Server Template Render -> Full HTML Response -> Repaint
  - SPA: Initial HTML/JS Load -> JS Runtime -> [[Client Side Routing]] -> API Fetch -> DOM Patch
- 데이터 흐름 (어디서 -> 어디로)
  - MPA: 링크 클릭 -> 새 URL 요청 -> 서버 렌더 -> 완성 HTML 전송 -> 브라우저가 전체 파싱
  - SPA: 라우트 변경 이벤트 -> 라우터가 컴포넌트 매핑 -> API 호출 -> 상태 저장소 갱신 -> 변경된 노드만 패치
- 상태 저장 위치
  - MPA: 요청 단위 상태는 서버 세션/DB 중심, 클라이언트 상태는 페이지 전환 시 대부분 소실
  - SPA: UI 상태는 브라우저 메모리(스토어, 컴포넌트 state)에 장시간 유지
- 네트워크 / 메모리 / 프로세스 관점
  - MPA: 요청 횟수는 많고 응답 단위는 크다(HTML 재전송). JS 런타임은 짧게 재시작되는 패턴.
  - SPA: 문서 요청은 적고 API 요청은 잦다. JS 런타임이 길게 유지되어 메모리 관리가 중요하다.
- 실제 동작 시나리오
  - 상품 목록 -> 상세 이동
  - MPA: 상세 URL로 전체 이동, 문서 전체 재생성
  - SPA: URL은 바뀌지만 문서는 유지, 상세 데이터만 받아 컴포넌트 재렌더

## 4. 활용
- MPA 활용: 콘텐츠 중심 사이트, 단순한 문서 네비게이션, 서버 렌더 중심 운영 조직.
- SPA 활용: 대시보드, 협업툴, 실시간 인터랙션이 많은 제품.
- 혼합 전략: 초기 표시와 SEO를 위해 [[SSR]] + [[Hydration]]을 결합한 하이브리드 구조.

## 5. 한계와 이후 발전
- MPA의 한계: 잦은 전체 리로드, 풍부한 상호작용 확장 시 비용 증가.
- SPA의 한계: 초기 번들 지연, 크롤링/메타 처리 복잡성, 장시간 런타임 메모리 누수 위험.
- 이후 발전: SPA의 UX를 유지하면서 초기 표시를 개선하기 위해 [[SSR]]과 [[Hydration]]이 발전했다.

## 5. 🔎 확장 질문
- ★★★★★ SPA에서 라우팅 상태와 서버 데이터 상태를 분리해야 하는 이유는 무엇인가?
- ★★★★☆ SSR과 Hydration의 경계에서 발생하는 불일치는 왜 생기는가?
- ★★★★☆ MPA에서 부분 갱신(AJAX)만으로 SPA 수준 UX를 만들기 어려운 이유는 무엇인가?

## 6. 🧠 이해 점검 퀴즈
- 단답형: SPA에서 페이지 전환 시 "전체 HTML 재요청" 대신 주로 무엇을 요청하는가?
- 서술형: MPA와 SPA의 렌더 경로를 네트워크와 메모리 관점으로 비교하라.
- 설계 이유: 왜 현대 프레임워크는 SPA 단독보다 SSR 결합 구조를 채택하는가?
