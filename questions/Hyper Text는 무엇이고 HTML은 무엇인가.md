# Hyper Text는 무엇이고 HTML은 무엇인가?

#질문

종이 문서를 읽을 때는 처음부터 끝까지 순서대로 읽는다. 그런데 실제 지식 탐색은 그렇지 않다. 어떤 단어를 읽다가 관련 개념으로 점프하고, 다시 원래 맥락으로 돌아오기를 반복한다. [[하이퍼텍스트]]는 바로 이 점프를 문서 구조 자체에 넣은 개념이다.

즉 하이퍼텍스트는 "선형 읽기"를 "비선형 탐색"으로 바꾼다. 문서 안에 링크를 심어 문서 간 이동 경로를 만든다. 웹은 이 아이디어를 네트워크 위로 올린 것이다.

[[HTML]]은 하이퍼텍스트 문서를 기계가 해석할 수 있게 표현하는 마크업 언어다. 핵심은 "내용의 의미를 태그로 구조화"하는 데 있다. 제목, 문단, 목록, 링크 같은 요소를 태그로 표시하면 브라우저가 문서 구조를 일관되게 해석할 수 있다.

![[assets/images/HTML-DOM-트리.svg]]

low-level에서는 브라우저가 HTML 문자열을 토큰으로 분해하고, 토큰을 규칙에 맞게 조립해 DOM 트리를 만든다. 이 DOM은 화면을 직접 그리는 픽셀이 아니라, 문서 구조를 나타내는 메모리 객체 그래프다. 이후 CSS/스크립트와 결합되며 실제 렌더링 결과가 결정된다.

이렇게 보면 하이퍼텍스트와 HTML의 관계가 명확하다. 하이퍼텍스트는 "문서를 연결하는 아이디어"이고, HTML은 "그 아이디어를 포함해 문서를 구조화하는 구현 언어"다. 아이디어와 구현을 구분해 보면, 왜 HTML이 단순 텍스트 포맷이 아니라 구조 언어인지 이해하기 쉽다.

한계도 있었다. HTML만으로는 표현(스타일)과 동작(인터랙션)을 깔끔히 분리하기 어렵다. 그래서 CSS와 JavaScript가 분화되며 웹의 역할이 확대된다. 결과적으로 HTML은 지금도 웹의 뼈대이며, 의미 구조를 선언하는 중심 언어로 남아 있다.

## 🔎 확장 질문

★★★★★ "의미론적 마크업"은 접근성과 검색 품질에 왜 영향을 주는가?

> [!important]
> 브라우저, 스크린리더, 검색 엔진은 태그 의미를 신호로 사용한다. 적절한 태그를 쓰면 문서 구조를 더 정확히 해석해 탐색성, 접근성, 인덱싱 품질이 개선된다.

★★★★☆ DOM은 왜 트리 구조여야 하는가?

> [!important]
> HTML 문서는 중첩된 계층 구조를 갖는다. 부모-자식 관계를 표현하기에 트리가 가장 자연스럽고, 부분 업데이트와 이벤트 전파 같은 연산도 계층 모델에서 효율적으로 정의된다.

★★★☆☆ 하이퍼텍스트 개념이 API 문서나 위키 시스템에서 특히 강력한 이유는 무엇인가?

> [!important]
> 개념 간 참조가 많을수록 비선형 탐색 가치가 커진다. 링크 기반 구조는 맥락 전환 비용을 낮춰 학습 속도를 높이고, 지식 그래프를 점진적으로 확장하게 만든다.

## 🧠 이해 점검 퀴즈

**Q1 (단답형)** HTML을 파싱한 결과 브라우저 메모리에 만들어지는 구조는 무엇인가?

> [!important]
> DOM 트리.

**Q2 (서술형)** 하이퍼텍스트와 HTML의 차이를 설명하라.

> [!important]
> 하이퍼텍스트는 문서 간 연결과 비선형 탐색이라는 개념이고, HTML은 그 개념을 포함해 문서 구조를 태그로 표현하는 구체적 마크업 언어다.

**Q3 (설계 의도)** HTML이 "표현"보다 "구조/의미"를 중심으로 설계된 이유는 무엇인가?

> [!important]
> 문서의 의미 구조를 표준화하면 다양한 브라우저/도구가 동일하게 해석할 수 있고, 표현 방식은 별도 계층(CSS)에서 유연하게 바꿀 수 있기 때문이다.
