# IaaS, PaaS, SaaS 의 차이와 활용은?

#질문

서비스를 운영하려면 결국 어딘가에서 서버를 돌리고, 저장소를 두고, 배포를 관리해야 한다. 예전에는 이 일을 직접 장비를 사고 설치하며 해결하는 경우가 많았다. 그런데 규모가 커질수록 "어디까지를 내가 책임지고 어디부터를 외부 서비스에 맡길 것인가"가 더 중요한 문제가 됐다. [[IaaS]], [[PaaS]], [[SaaS]]는 이 책임 분배를 설명하는 대표적인 층위다.

[[IaaS]]는 인프라 자체를 가상화해 빌려주는 모델이다. 운영체제, 런타임, 배포 방식은 내가 직접 다루되, 물리 장비와 네트워크 일부는 공급자가 맡는다. [[PaaS]]는 그보다 한 단계 더 올라가서 애플리케이션을 올릴 실행 환경까지 제공한다. [[SaaS]]는 사용자가 완성된 소프트웨어를 서비스 형태로 쓰는 모델이다.

![[assets/images/클라우드-컴퓨팅-구조.svg]]

비유하면 IaaS는 빈 사무실을 임대하는 것이고, PaaS는 책상과 전기, 인터넷이 이미 갖춰진 공유 오피스를 쓰는 것에 가깝다. SaaS는 아예 필요한 업무 소프트웨어를 구독해 쓰는 상황과 비슷하다. 직접 통제권은 아래층으로 갈수록 커지지만, 관리 부담도 함께 커진다.

이 구분은 단순한 용어 암기가 아니라 운영 철학의 차이다. 예를 들어 프론트엔드 팀이 정적 사이트를 배포할 때, S3와 CDN을 직접 조합하면 IaaS 쪽에 가깝고, Vercel이나 Netlify 같은 배포 플랫폼을 쓰면 PaaS 성격이 강하다. 디자인 협업 툴이나 분석 툴을 구독해 사용하는 것은 SaaS에 가깝다.

중요한 포인트는 책임이 사라지는 것이 아니라 이동한다는 점이다. [[Shared Responsibility Model]]을 보면 공급자가 맡는 영역이 늘어나도, 애플리케이션 코드 품질, 접근 제어, 비밀값 관리, 비용 통제는 여전히 사용자 책임이 남는다. 그리고 서비스에 깊이 의존할수록 [[Vendor Lock in]] 가능성도 커진다.

결국 IaaS, PaaS, SaaS의 차이는 기술 스택이 아니라 통제권과 운영 부담의 균형 문제다. 어떤 모델이 더 우월하다기보다, 우리 팀이 어디에 시간을 쓰고 싶은지를 결정하는 선택에 가깝다.

---

## 프론트엔드 개발자로써 이 내용을 활용할때 주의할 점

프론트엔드도 배포와 운영에서 예외가 아니다. 빌드 산출물, 서버리스 함수, 이미지 최적화, 캐시 무효화까지 포함하면 어떤 플랫폼을 쓰느냐가 개발 방식 자체를 바꾼다.

실제 활용 단계에서는 배포 속도, 프리뷰 환경, 로그 접근성, 커스텀 인프라 필요성, 락인 위험을 함께 봐야 한다. 편한 플랫폼을 고르는 것과, 장기적으로 통제권을 유지하는 것은 같은 문제가 아니다.

## 🔎 확장 질문

★★★★★ Shared Responsibility Model은 프론트엔드 배포에서도 왜 중요한가?

> [!important]
> 플랫폼이 인프라를 대신 관리해도, 코드 취약점, 비밀값 노출, 잘못된 캐시 정책은 여전히 사용자 책임이다. 운영을 맡겼다고 보안과 품질 책임까지 사라지지는 않는다.

★★★★☆ Vendor Lock in은 어떤 상황에서 실제 비용으로 돌아오는가?

> [!important]
> 특정 빌드 규약, 서버리스 인터페이스, 이미지 파이프라인, CI 흐름에 강하게 종속되면 이전 비용이 커진다. 초기 생산성이 높아도 장기 유연성을 잃을 수 있다.

★★★☆☆ 작은 팀은 왜 PaaS나 SaaS를 선호하는 경우가 많은가?

> [!important]
> 인프라 운영보다 제품 개발 속도가 더 중요할 때가 많기 때문이다. 반복 작업을 줄여 핵심 기능에 집중하기 쉽다.

## 🧠 이해 점검 퀴즈

**Q1 (단답형)** 완성된 소프트웨어를 구독 형태로 사용하는 모델은 무엇인가?

> [!important]
> SaaS.

**Q2 (서술형)** IaaS와 PaaS의 책임 분배 차이를 설명하라.

> [!important]
> IaaS는 가상 인프라를 제공하고 운영체제와 런타임, 배포 관리의 상당 부분을 사용자가 맡는다. PaaS는 실행 환경과 배포 플랫폼까지 더 많이 제공해 운영 부담을 줄인다.

**Q3 (설계 의도)** 클라우드 서비스는 왜 이런 층위별 모델로 발전했는가?

> [!important]
> 팀마다 원하는 통제권과 감당 가능한 운영 부담이 다르기 때문이다. 하드웨어부터 완성 소프트웨어까지 책임을 여러 층으로 나눠 선택할 수 있게 했다.
