# 초기의 웹페이지는 어떤 형태였고 웹브라우저는 무슨 역할인가?

#질문

초기의 웹페이지는 지금처럼 화려한 앱 화면이 아니었다. 학술 문서에 가까운 텍스트 중심 페이지였고, 제목과 본문, 링크가 핵심이었다. 이미지를 넣을 수는 있었지만 배치와 스타일 제어가 제한적이라, 정보 전달이 우선인 "읽기 페이지"에 가까웠다.

당시 [[웹 브라우저]]의 역할은 단순 앱 런처가 아니라 번역기였다. 서버가 보내준 HTML 텍스트를 사람이 볼 수 있는 화면으로 바꾸는 해석기 역할을 맡았다. 비유하면, 브라우저는 악보(HTML)를 받아 소리(화면)로 연주하는 연주자다. 악보 자체는 텍스트지만 연주 결과는 시각적 결과물이다.

![[assets/images/웹-최초-웹사이트-프로젝트.png]]

low-level 동작은 지금도 큰 틀은 같다. 브라우저가 URL을 해석해 서버에 요청하고, 응답으로 받은 HTML을 파싱해 [[DOM]]을 만든다. 초창기에는 여기서 끝나는 경우가 많았다. 문서를 렌더링하고 링크를 클릭하면 새 문서를 다시 받아 전체를 그렸다. 상태를 오래 유지하는 애플리케이션 개념보다 문서 이동 개념이 중심이었다.

브라우저가 중요한 이유는 표준 해석 창구이기 때문이다. 서버는 "표준 문서"를 주고, 브라우저는 그 표준을 해석해 사용자 경험으로 만든다. 이 경계가 명확해야 웹이 플랫폼으로 성장할 수 있다. 서버는 내용과 데이터에 집중하고, 브라우저는 표현과 상호작용을 담당하는 방향으로 진화했기 때문이다.

한계도 분명했다. 초기 웹페이지는 표현력이 낮고, 사용자 상호작용이 제한적이었다. 그래서 CSS가 도입돼 표현 제어가 강화되고, JavaScript가 도입돼 사용자 이벤트에 반응하는 동작이 가능해졌다. 초기 웹페이지는 단순했지만, 그 단순함 덕분에 웹의 공통 기반이 빠르게 자리 잡았다.

## 🔎 확장 질문

★★★★★ 브라우저 간 렌더링 차이는 왜 생기고, 표준은 이를 어떻게 줄여왔는가?

> [!important]
> 브라우저마다 렌더링 엔진 구현이 달라 같은 HTML/CSS라도 결과가 달라질 수 있다. W3C/WHATWG 표준과 브라우저 벤더의 호환성 테스트 축적이 이 차이를 줄이는 핵심 메커니즘이다.

★★★★☆ 초기 브라우저의 "문서 뷰어" 성격이 오늘날 웹앱 구조에 어떤 흔적을 남겼는가?

> [!important]
> URL 기반 네비게이션, 하이퍼링크 중심 탐색, 문서 리소스 로딩 모델이 여전히 웹의 기본 뼈대다. SPA 같은 현대 구조도 결국 브라우저의 문서/리소스 모델 위에서 동작한다.

★★★☆☆ 브라우저가 보안 경계를 맡는 이유는 무엇인가?

> [!important]
> 브라우저는 외부 네트워크 콘텐츠를 로컬 환경에서 실행 가능한 형태로 해석한다. 이 지점이 공격 표면이 되므로 동일 출처 정책, 샌드박스, 권한 모델 같은 보안 경계가 브라우저에 집중된다.

## 🧠 이해 점검 퀴즈

**Q1 (단답형)** 브라우저가 HTML을 파싱해 만드는 트리 구조의 이름은?

> [!important]
> DOM(Document Object Model).

**Q2 (서술형)** 초기 웹페이지가 "문서 중심"이었다는 말의 의미를 설명하라.

> [!important]
> 페이지 단위로 서버에서 문서를 받아 읽고 링크로 다음 문서로 이동하는 구조가 중심이었다는 뜻이다. 앱처럼 복잡한 상태 유지/부분 갱신보다 문서 탐색과 정보 전달이 핵심이었다.

**Q3 (설계 의도)** 브라우저가 서버와 분리된 독립 클라이언트로 설계된 이유는 무엇인가?

> [!important]
> 서버는 콘텐츠 제공에 집중하고, 클라이언트는 사용자 환경에 맞춘 렌더링과 상호작용을 담당하도록 책임을 분리하기 위해서다. 이 분리가 확장성과 이식성을 높였다.
