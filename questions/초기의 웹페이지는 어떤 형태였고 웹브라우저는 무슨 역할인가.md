# 초기의 웹페이지는 어떤 형태였고 웹브라우저는 무슨 역할인가?

#질문

초기의 웹페이지는 오늘날의 앱 화면과는 꽤 달랐다. 상호작용이 풍부한 대시보드라기보다, 문서와 링크 중심의 전자 페이지에 가까웠다. 텍스트가 중심이고, 이미지는 제한적이며, 페이지를 이동할 때마다 새 문서를 읽는 흐름이 자연스러웠다.

이런 환경에서 [[웹 브라우저]]의 역할은 단순한 뷰어가 아니었다. 브라우저는 서버에서 내려받은 문서를 해석해 사람이 읽을 수 있는 화면으로 바꾸는 번역기이자, 링크를 따라 다른 문서로 이동하게 해 주는 탐색기였다. 즉 웹의 표준 규약을 실제 사용자 경험으로 바꾸는 실행기였다.

![[assets/images/웹-최초-웹사이트-프로젝트.png]]

브라우저는 HTML을 파싱해 [[DOM]] 구조를 만들고, 스타일 규칙이 있다면 적용하고, 이후에는 [[렌더링 엔진]]을 통해 화면을 그린다. 초창기에는 표현과 동작이 단순했기 때문에 오늘날처럼 복잡한 최적화가 덜했지만, 기본 역할은 이미 정해져 있었다. 문서를 가져오고, 해석하고, 그려 주는 것 말이다.

비유하면 브라우저는 PDF 리더와 네비게이터와 번역기를 한 몸에 합친 도구였다. 문서 내용을 그대로 보여 주는 데서 끝나지 않고, 하이퍼링크라는 이동 수단과 함께 사용자에게 "다음 문서로 이어지는 길"을 제공했다.

웹이 점차 복잡해지면서 브라우저는 더 많은 책임을 떠안게 됐다. CSS 처리, JavaScript 실행, 보안 정책, 캐시, 네트워크 최적화, 개발자 도구까지 포함하게 된 것이다. 하지만 출발점을 기준으로 보면 브라우저의 본질은 여전히 같다. 웹의 공통 규약을 실제 사용자 인터페이스로 변환하는 실행 환경이다.

결국 초기 웹페이지가 단순했기 때문에 브라우저의 핵심 역할도 선명하게 보인다. 브라우저는 "사이트를 열어 주는 앱"이 아니라, 웹이라는 표준 세계를 사람이 읽을 수 있게 실시간으로 해석하는 엔진이다.

## 프론트엔드 개발자로써 이 내용을 활용할때 주의할 점

브라우저를 단순한 실행 대상이라고만 보면 안 된다. 브라우저는 네트워크, 파싱, 렌더링, 보안 정책을 동시에 담당하는 플랫폼이기 때문에, 프론트엔드 코드는 항상 그 제약 위에서 돌아간다.

실제 활용 단계에서는 브라우저 호환성, 기본 동작, 시맨틱 마크업, 렌더링 비용을 함께 고려해야 한다. 프레임워크가 모든 것을 추상화해 주는 것처럼 보여도 최종 실행 주체는 여전히 브라우저다.
