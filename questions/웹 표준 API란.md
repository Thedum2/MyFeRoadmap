# 웹 표준 API란?
#질문

## 1. 등장 배경
초기 웹은 문서를 보여주는 플랫폼이었다. 링크를 누르면 서버가 HTML을 보내고, 브라우저는 그 문서를 렌더링했다. 문제는 웹이 문서 소비에서 애플리케이션 실행 환경으로 바뀌면서 생겼다. 개발자는 파일 접근, 네트워크 요청, 타이머, 화면 업데이트 같은 기능이 필요했지만, 브라우저마다 구현이 달라 코드가 자주 깨졌다.

같은 기능을 만들었는데 브라우저마다 동작이 다르면 제품을 운영하기 어렵다. 그래서 "브라우저 기능을 공통 인터페이스로 노출"하는 합의가 필요해졌다. 이 합의가 웹 표준 API의 출발점이다. 핵심은 특정 벤더 기능이 아니라, 여러 엔진에서 동일하게 호출 가능한 규격을 만드는 것이다.

## 2. 구조적 전환
웹 표준 API는 단순 라이브러리가 아니다. 브라우저 엔진이 제공하는 기능을 자바스크립트에서 접근 가능한 객체 모델로 노출한 것이다. 문서 조작은 [[DOM]], 네트워크는 [[Fetch API]], 이벤트 처리와 비동기 실행은 [[이벤트 루프]]와 연동된다.

표준화는 보통 명세 문서에서 시작된다. API 시그니처, 타입, 예외, 동작 규칙은 [[Web IDL]]로 기술되고, 브라우저 구현체는 이를 기반으로 동일한 표면을 맞춘다. 그래서 개발자는 엔진 내부 구현을 몰라도 "같은 API 이름과 계약"으로 코드를 작성할 수 있다.

## 3. 내부 동작 원리 (Low-Level)
브라우저에서 `fetch()`를 호출하면 자바스크립트 엔진이 즉시 네트워크 I/O를 직접 수행하지는 않는다. 호출은 브라우저의 네트워크 스택으로 위임되고, 자바스크립트 스레드는 논블로킹 상태를 유지한다. 요청 완료 신호는 태스크/마이크로태스크 큐를 거쳐 이벤트 루프가 처리하고, Promise 콜백이 재개된다.

`document.querySelector()` 같은 DOM API 호출은 렌더러 프로세스의 DOM 트리 객체에 접근해 노드를 찾고, 변경이 생기면 스타일 계산과 레이아웃, 페인트 파이프라인이 뒤따른다. 즉 API 한 줄은 내부적으로 파싱 트리, 스타일 엔진, 컴포지터 단계까지 연결된다.

보안도 API 계약의 일부다. 스크립트가 임의 출처의 민감 자원에 바로 접근하지 못하도록 [[동일 출처 정책]]이 기본 경계를 만든다. 다른 출처와 통신이 필요하면 [[CORS]] 헤더 기반 협상이 필요하다.

> [!important]
> 웹 표준 API의 low-level 본질은 "JS 호출과 브라우저 서브시스템 사이의 안정된 경계"다. 표면은 단순하지만 내부는 멀티 프로세스, 큐, 렌더링 파이프라인으로 구성된다.

## 4. 활용
실무에서 웹 표준 API는 프레임워크보다 더 아래층에서 계속 사용된다. React나 Vue를 써도 결국 이벤트 등록, 네트워크 요청, 스토리지 접근, URL 제어는 브라우저 표준 API 위에서 동작한다.

예를 들어 대시보드에서 필터를 바꾸면 Fetch API로 데이터 요청을 보내고, 응답 Promise가 resolve되면 DOM 업데이트가 일어난다. 이때 히스토리 API로 URL 상태를 동기화하면 새로고침 후에도 같은 화면을 복원할 수 있다.

웹뷰, 데스크톱 래퍼, 모바일 브라우저에서도 표준 API를 중심으로 설계하면 이식성이 높다. 브라우저별 분기 코드는 줄고, 테스트와 유지보수의 기준점이 명확해진다.

## 5. 한계와 이후 발전
표준 API가 있다고 해도 모든 문제가 사라지지는 않는다. 브라우저 버전 차이, 실험 기능 플래그, 성능 특성 차이 때문에 동작은 같아도 체감 성능이 다를 수 있다. 또 API가 늘어날수록 권한 모델과 프라이버시 경계도 복잡해진다.

그래서 최근 발전은 두 방향으로 진행된다. 하나는 명세 정밀화다. 에러 케이스, 타이밍, 보안 제약을 더 엄격히 정의해 상호운용성을 높인다. 다른 하나는 점진적 기능 도입이다. 새 API를 바로 강제하지 않고 폴리필, 기능 감지, 단계적 배포로 생태계 충격을 줄인다. 웹 표준 API는 "한 번 완성"이 아니라, 호환성과 혁신의 균형을 계속 조정하는 프로세스다.

## 6. 🔎 확장 질문
- ★★★★★ 왜 웹 표준 API는 벤더 독자 기능보다 느리게 진화해도 가치가 큰가?
> [!important]
> 속도보다 상호운용성과 예측 가능성이 제품 리스크를 크게 줄이기 때문이다. 표준 API는 장기 유지보수와 브라우저 간 일관성을 보장해 총비용을 낮춘다.

- ★★★★☆ Fetch API가 Promise 기반으로 설계된 이유는 무엇인가?
> [!important]
> 네트워크 I O를 논블로킹으로 처리하고 이벤트 루프와 자연스럽게 결합하기 위해서다. 콜백 중첩을 줄이고 오류 전파를 구조화하기에도 유리하다.

- ★★★☆☆ 동일 출처 정책과 CORS를 분리해 둔 설계 이유는 무엇인가?
> [!important]
> 기본값은 차단으로 안전 경계를 만들고, 필요한 경우에만 서버가 명시적으로 허용하도록 하기 위해서다. 보안 기본값과 서비스 유연성을 동시에 맞추는 구조다.

## 7. 🧠 이해 점검 퀴즈
- 단답형: 브라우저 API 시그니처와 타입 규약을 기술하는 표준 기술 언어는 무엇인가?
> [!important]
> Web IDL

- 서술형: `fetch()` 호출 이후 응답 처리 콜백이 실행될 때까지의 과정을 이벤트 루프 관점에서 설명하라.
> [!important]
> fetch 호출은 브라우저 네트워크 계층으로 위임되고 자바스크립트 실행은 계속된다. 응답이 준비되면 완료 이벤트가 큐에 들어가고, 이벤트 루프가 이를 처리해 Promise를 settle한다. 이후 마이크로태스크 큐의 then/catch 콜백이 실행된다.

- 왜 이런 설계를 했는가?: 웹 표준 API를 "명세 + 구현 + 테스트" 체계로 운영하는 이유는 무엇인가?
> [!important]
> 문서만으로는 해석 차이가 생기기 쉽기 때문이다. 명세로 계약을 정의하고, 구현으로 현실성을 검증하고, 상호운용 테스트로 동작 일치를 강제해야 다중 브라우저 생태계가 안정된다.
