# 웹이란 무엇인가?
#질문

## 1. 등장 배경
처음의 인터넷은 연구자와 군 통신망을 잇는 [[인터넷]] 인프라였다. 컴퓨터끼리 데이터를 주고받을 수는 있었지만, 사람이 정보를 찾고 읽는 경험은 매우 거칠었다. FTP로 파일을 내려받고, 텔넷으로 원격 접속하고, 서비스마다 명령어를 따로 익혀야 했다. 정보는 많아졌지만 접근 방식이 제각각이라 "연결은 되었는데 이해는 어려운" 상태가 오래 지속되었다.

팀 버너스리가 제안한 [[월드 와이드 웹]]의 핵심은 간단했다. 문서를 문서와 연결하고, 어디서나 같은 방식으로 열어보게 하자. 이 선택이 "네트워크"를 "지식 공간"으로 바꿨다.

## 2. 구조적 전환
웹의 구조적 전환은 세 가지 약속으로 정리된다. 자원의 주소를 [[URL]]로 통일하고, 전송 규약을 [[HTTP]]로 통일하고, 표현 형식을 HTML로 통일한 것이다. 이 조합은 서비스마다 다른 클라이언트를 강요하던 시대를 끝냈다.

여기에 [[클라이언트 서버 모델]]을 채택하면서 역할이 분리됐다. 브라우저는 요청하고 렌더링한다. 서버는 요청을 해석하고 응답한다. 기능 분리는 확장성과 운영 효율을 크게 올렸지만, 동시에 지연 시간과 병목이라는 새로운 문제도 함께 가져왔다.

## 3. 내부 동작 원리 (Low-Level)
브라우저 주소창에 URL을 입력하면 먼저 [[DNS]] 조회가 일어난다. 도메인 이름을 IP 주소로 바꾸는 단계다. 그다음 전송 계층에서 [[TCP IP|TCP/IP]] 연결을 맺고, TLS가 필요한 경우 암호화 핸드셰이크를 추가한다. 이 시점까지가 "서버를 안전하게 찾고 연결하는" 과정이다.

연결이 준비되면 브라우저는 HTTP 요청 메시지를 보낸다. 요청 줄, 헤더, 필요 시 본문이 포함된다. 서버는 상태 코드, 헤더, 본문으로 응답한다. HTML을 받은 브라우저는 파싱을 시작하고, CSS와 JavaScript 같은 하위 리소스를 다시 요청한다. 즉, 한 번의 페이지 열기 안에서 여러 HTTP 왕복이 연쇄적으로 발생한다.

이때 성능을 좌우하는 핵심은 왕복 횟수와 전송량이다. HTTP 1.1은 직렬 대기 문제가 컸고, HTTP 2는 멀티플렉싱으로 연결 하나에서 여러 스트림을 동시에 처리하도록 개선했다. 웹은 "문서 열람 기술"에서 출발했지만, 내부적으로는 패킷, 큐, 혼잡 제어, 캐시 정책이 맞물려 동작하는 분산 시스템에 가깝다.

## 4. 활용
웹은 문서 공유를 넘어 애플리케이션 실행 환경이 되었다. 전자상거래, 협업 도구, 스트리밍, SaaS 모두 브라우저 위에서 동작한다. 예를 들어 온라인 지도 서비스는 초기 HTML 전달 후 타일 이미지와 API 데이터를 비동기로 계속 받아 화면을 갱신한다.

중요한 점은 웹이 "배포 비용"을 극단적으로 낮췄다는 것이다. 사용자는 설치 없이 URL만 열면 된다. 이 특성 덕분에 서비스 실험과 글로벌 확장이 다른 플랫폼보다 빠르게 일어났다.

## 5. 한계와 이후 발전
웹의 기본 철학은 느슨한 결합과 호환성이다. 덕분에 폭발적으로 성장했지만, 실시간성·고성능 계산·오프라인 신뢰성에서는 네이티브 앱 대비 약점이 있었다. 또 브라우저 보안 모델은 안전을 높이는 대신 권한 접근을 제한해 기능 구현을 복잡하게 만든다.

그래서 이후 발전은 "호환성을 지키면서 성능을 끌어올리는" 방향으로 진행됐다. HTTP 2와 HTTP 3, 서비스 워커, WebAssembly, 스트리밍 렌더링 같은 기술이 그 연장선이다. 웹은 여전히 단일 기술이 아니라, 표준의 합의와 점진적 개선으로 진화하는 생태계다.

## 6. 🔎 확장 질문
- ★★★★★ 인터넷과 웹은 왜 같은 말이 아닌가?
> [!important]
> 인터넷은 네트워크 인프라 자체이고, 웹은 그 위에서 동작하는 하이퍼텍스트 서비스다. 이메일이나 게임 트래픽은 인터넷을 쓰지만 웹은 아닐 수 있다.

- ★★★★☆ URL 설계가 왜 웹 확장성에 결정적이었는가?
> [!important]
> 자원을 전역 주소로 식별하면 클라이언트와 서버가 느슨하게 결합된다. 주소 규칙만 지키면 구현이 달라도 상호 운용이 가능해진다.

- ★★★★☆ HTTP 2가 체감 성능을 개선한 핵심 이유는 무엇인가?
> [!important]
> 하나의 연결에서 다중 스트림을 병렬 처리해 대기 시간을 줄였기 때문이다. 리소스가 많은 페이지에서 직렬 병목이 크게 완화된다.

## 7. 🧠 이해 점검 퀴즈
- 단답형: 웹 요청에서 도메인 이름을 IP로 바꾸는 단계의 이름은?
> [!important]
> DNS 조회.

- 서술형: 브라우저가 첫 화면을 그릴 때 네트워크 왕복이 여러 번 필요한 이유를 설명하라.
> [!important]
> 초기 HTML만으로는 스타일과 동작 정보가 완성되지 않기 때문이다. HTML 파싱 중 CSS, JavaScript, 이미지 등 추가 리소스 URL이 발견되고, 각 리소스마다 별도 요청/응답이 발생한다.

- 왜 이런 설계를 했는가?: 웹이 기능적으로 불리해도 브라우저 기반 배포 모델을 유지한 이유는?
> [!important]
> 설치 없이 즉시 접근 가능한 배포 특성이 네트워크 효과를 만들기 때문이다. 호환성과 도달성이 성능 손실보다 더 큰 제품/비즈니스 가치를 제공했다.
