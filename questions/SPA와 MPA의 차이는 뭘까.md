# SPA와 MPA의 차이는 뭘까?

#질문

웹이 처음 대중화되던 시절을 떠올리면, 페이지 하나를 열 때마다 브라우저는 매번 새 종이를 통째로 갈아끼우는 방식으로 동작했다. 링크를 누르면 서버가 완성된 HTML 문서를 다시 보내고, 브라우저는 화면 전체를 다시 그렸다. 이 방식이 바로 [[MPA]]의 기본 감각이다. 문서 단위로 이동하고, 이동할 때마다 서버가 새 페이지를 책임진다.

이 방식은 단순하고 강력하다. 서버가 화면을 완성해서 내려주니 초기 표시가 빠르고, 검색 엔진도 HTML을 바로 읽기 쉽다. 하지만 사용자가 앱처럼 빠른 전환을 기대하기 시작하면서 불편이 드러난다. 탭을 누를 때마다 화면이 깜빡이고, 공통 레이아웃까지 반복해서 다시 받는다. 마치 건물 안에서 방 하나만 옮기고 싶은데, 매번 건물 정문으로 나가 다시 입장하는 느낌이다.

그래서 나온 선택이 [[SPA]]다. SPA는 첫 진입에서 큰 틀을 한 번 받고, 이후 화면 전환은 브라우저 안에서 자바스크립트가 처리한다. 즉, 문서 전체를 다시 받는 대신 필요한 데이터만 API로 받아 화면 조각만 교체한다. 이 렌더링 전략을 [[Client Side Rendering]]이라고 부른다.

![[assets/images/SPA-structure-wikimedia.png]]

low-level로 보면 차이는 더 명확하다. MPA는 사용자 액션마다 브라우저가 새 HTTP 요청으로 문서를 받고, HTML 파싱, CSSOM 구성, 렌더 트리 생성, 페인트 과정을 페이지 단위로 다시 수행한다. 반면 SPA는 초기 번들 로딩 이후에는 JSON 같은 데이터 응답을 받고, 프런트엔드 런타임이 상태를 바꾼 뒤 필요한 DOM만 갱신한다. 전체 교체 대신 부분 갱신이 기본 전략이다.

그렇다면 SPA가 항상 이기느냐 하면 그렇지 않다. SPA는 초기 번들이 크면 첫 로딩이 느려질 수 있고, 자바스크립트 실행 비용이 커진다. 또 SEO나 소셜 미리보기 대응을 위해 서버에서 HTML을 먼저 그려주는 [[Server Side Rendering]]이나 [[Hydration]] 같은 보완 전략이 자주 필요하다. 결국 SPA는 "전환 속도"를 얻는 대신 "초기 로딩, 런타임 복잡도"를 지불하는 구조다.

MPA도 발전했다. 전통적인 서버 렌더링 위에 부분 갱신을 얹거나, 페이지 단위 라우팅은 유지하되 캐싱과 프리페치를 강화해 체감 속도를 끌어올린다. SPA 역시 [[Code Splitting]]으로 초기 번들을 쪼개고, SSR과 결합해 첫 화면 표시를 개선한다. 요즘 프레임워크가 SPA와 MPA의 중간 지대를 넓게 제공하는 이유가 여기에 있다.

실무에서는 서비스 성격으로 판단하면 된다. 백오피스처럼 화면 전환이 잦고 상호작용이 많은 제품은 SPA 성향이 유리하다. 반대로 콘텐츠 중심 사이트, 검색 유입이 중요한 문서형 서비스는 MPA 또는 SSR 중심 구성이 더 안정적이다. 핵심은 "어느 방식이 더 현대적인가"가 아니라, 병목이 네트워크 왕복인지 초기 로딩인지 운영 복잡도인지부터 먼저 확인하는 것이다.

## 🔎 확장 질문

★★★★★ SSR과 Hydration은 SPA의 어떤 약점을 보완하기 위해 등장했는가?

> [!important]
> SPA는 초기 번들 다운로드와 자바스크립트 실행이 끝나기 전까지 의미 있는 화면이 늦게 보일 수 있다. SSR은 서버가 먼저 HTML을 만들어 첫 화면을 빠르게 보여주고, Hydration은 그 HTML에 클라이언트 이벤트를 다시 연결해 상호작용을 복구한다. 즉, 첫 표시 성능과 상호작용성을 분리해 최적화하는 전략이다.

★★★★☆ MPA에서도 앱처럼 빠른 화면 전환을 만들 수 있는가?

> [!important]
> 가능하다. 전체 문서 재로딩 대신 프리페치, 캐시 제어, 부분 HTML 교체(예: HTMX 계열), 스트리밍 SSR을 조합하면 전환 체감 속도를 크게 개선할 수 있다. 다만 라우팅과 상태 동기화의 책임 분배가 SPA와 다르므로 아키텍처 기준을 명확히 잡아야 한다.

★★★☆☆ 모바일 저사양 환경에서는 SPA와 MPA 중 어떤 선택이 더 유리한가?

> [!important]
> 저사양 디바이스에서는 자바스크립트 파싱/실행 비용이 체감 성능을 크게 악화시킬 수 있어 MPA나 SSR 중심이 유리한 경우가 많다. 반대로 오프라인 동작, 복잡한 UI 상태 유지가 핵심이라면 SPA가 필요할 수 있다. 결국 CPU 비용과 네트워크 비용 중 무엇이 더 큰 병목인지 측정으로 판단해야 한다.

## 🧠 이해 점검 퀴즈

**Q1 (단답형)** SPA에서 화면 전환 시 일반적으로 서버에서 다시 내려받지 않는 것은 무엇인가?

> [!important]
> 페이지 전체 HTML 문서.

**Q2 (서술형)** MPA의 페이지 단위 재요청 구조가 사용자 경험에 주는 장단점을 설명하라.

> [!important]
> 장점은 서버가 완성된 HTML을 제공하므로 초기 표시, SEO, 캐시 전략이 단순하고 안정적이라는 점이다. 단점은 이동 시 전체 재렌더링이 반복되어 전환 체감이 느려질 수 있고, 공통 레이아웃까지 중복 전송되는 비효율이 생긴다는 점이다.

**Q3 (설계 의도)** 왜 현대 프레임워크들은 SPA와 MPA를 이분법으로 나누지 않고 혼합 전략(SSR, Hydration, 코드 분할)을 제공하는가?

> [!important]
> 실제 서비스 병목이 하나로 고정되지 않기 때문이다. 어떤 페이지는 초기 표시가 중요하고, 어떤 화면은 상호작용 밀도가 높다. 혼합 전략은 초기 표시 성능, 전환 속도, SEO, 운영 복잡도 사이의 트레이드오프를 화면 단위로 조정하게 해 준다.
