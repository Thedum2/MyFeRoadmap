# SPA와 MPA의 차이는 뭘까?

#질문

웹이 문서 중심이던 시절에는 페이지를 이동할 때마다 새 HTML 문서를 받아 오는 방식이 자연스러웠다. 뉴스 목록에서 상세 페이지로, 다시 홈으로 갈 때 서버가 각 경로마다 완성된 문서를 반환하는 구조다. 이것이 [[MPA]]의 기본 감각이다. 여러 개의 페이지가 있고, 이동할 때마다 브라우저가 새 문서를 여는 셈이다.

서비스가 애플리케이션처럼 복잡해지자 다른 요구가 생겼다. 메뉴를 옮겨 다녀도 헤더와 사이드바는 그대로 두고, 화면 일부만 바꾸며 상태를 이어 가고 싶어졌다. 이때 등장한 방식이 [[SPA]]다. 처음에는 비교적 얇은 HTML과 JavaScript 번들을 받고, 이후 화면 전환은 클라이언트가 관리한다.

![[assets/images/SPA-MPA-comparison-wikimedia.svg]]

비유하면 MPA는 방마다 문 전체를 열고 들어가는 건물이고, SPA는 큰 로비 안에서 칸막이와 조명을 바꿔 장면을 전환하는 공간에 가깝다. MPA는 문서 단위 이동이 명확하고, SPA는 앱처럼 이어지는 경험을 만들기 쉽다.

내부 동작도 다르다. MPA에서는 경로가 바뀔 때 서버가 보통 새로운 HTML을 반환하고, 브라우저는 HTML 파싱부터 [[DOM]] 구성, CSS 적용, 스크립트 실행을 다시 시작한다. 반면 SPA는 라우터가 경로 변화를 감지하고, 필요한 데이터와 컴포넌트를 불러와 기존 DOM 일부를 갱신한다. 초기 진입은 무겁더라도 이후 전환은 빠르게 느껴질 수 있다.

![[assets/images/SPA-structure-wikimedia.png]]

그래서 선택 기준도 분명해진다. 콘텐츠 중심, SEO 의존도가 높고 문서 단위 이동이 자연스러운 서비스는 MPA나 [[Server Side Rendering]] 조합이 유리할 수 있다. 반대로 대시보드, 협업 도구, 설정 화면처럼 상태 유지와 잦은 상호작용이 중요한 서비스는 SPA나 [[Client Side Rendering]] 중심 구조가 잘 맞는다.

물론 경계는 점점 흐려졌다. 현대 프레임워크는 SPA 경험 위에 SSR, [[Hydration]], [[Code Splitting]]을 얹는다. 즉 "SPA냐 MPA냐"는 이분법이라기보다, 초기 응답과 이후 상호작용의 책임을 서버와 클라이언트 사이에 어떻게 나눌지에 대한 설계 문제다.

결국 SPA와 MPA의 차이는 화면을 몇 번 새로 그리느냐보다, 라우팅과 렌더링의 중심이 어디 있느냐에 있다. 문서가 이동의 주인공이면 MPA에 가깝고, 클라이언트 상태가 이동의 주인공이면 SPA에 가깝다.

---

## 프론트엔드 개발자로써 이 내용을 활용할때 주의할 점

SPA를 선택했다고 항상 빠른 것은 아니다. 초기 번들이 크면 첫 진입이 무거워지고, MPA를 선택했다고 항상 단순한 것도 아니다. 라우팅, 캐싱, 상태 지속성 요구를 같이 봐야 한다.

실제 활용 단계에서는 SEO, 초기 로딩, 내비게이션 체감 속도, 에러 복구, 캐시 전략을 함께 비교해야 한다. 특히 SPA에서는 코드 분할과 데이터 패칭 전략, MPA에서는 경로별 캐싱과 서버 렌더링 비용이 핵심 포인트다.

## 🔎 확장 질문

★★★★★ SSR과 Hydration은 SPA와 MPA의 장단점을 어떻게 섞으려는가?

> [!important]
> 서버가 초기 HTML을 빠르게 보여 주고, 클라이언트가 이후 상호작용을 이어받는 구조다. 초기 체감 속도와 앱 같은 경험을 동시에 얻으려는 절충안이다.

★★★★☆ Code Splitting은 SPA에서 왜 특히 중요해지는가?

> [!important]
> SPA는 초기 진입 시 클라이언트가 감당해야 할 코드가 많아지기 쉽다. 화면 단위로 코드를 나누지 않으면 첫 로딩 비용이 급격히 커진다.

★★★☆☆ MPA는 현대 웹에서도 여전히 유효한가?

> [!important]
> 그렇다. 문서 중심 서비스, 서버 캐싱이 강한 서비스, 단순한 상호작용 중심 서비스에서는 여전히 단순하고 견고한 선택이 될 수 있다.

## 🧠 이해 점검 퀴즈

**Q1 (단답형)** 경로 이동 때마다 서버가 새 HTML 문서를 반환하는 구조에 더 가까운 방식은 무엇인가?

> [!important]
> MPA.

**Q2 (서술형)** SPA와 MPA의 렌더링 책임 분배 차이를 설명하라.

> [!important]
> MPA는 경로별 문서 생성과 이동의 중심이 서버에 더 가깝고, SPA는 초기 진입 후 화면 전환과 상태 유지의 중심이 클라이언트에 더 가깝다.

**Q3 (설계 의도)** 현대 프레임워크가 순수 SPA나 순수 MPA 대신 혼합 전략을 택하는 이유는 무엇인가?

> [!important]
> 초기 응답 속도, SEO, 상호작용 품질, 개발 생산성 요구를 동시에 만족시키기 어렵기 때문이다. 그래서 서버와 클라이언트의 책임을 상황에 맞게 나눠 가진다.
