# 웹은 무엇이고 왜 탄생하였는가?

#질문

웹을 이해하려면 먼저 1980~1990년대 연구소 풍경을 떠올리면 쉽다. 컴퓨터는 이미 있었고 네트워크도 있었다. 문제는 정보가 섬처럼 흩어져 있었다는 점이다. 서로 다른 컴퓨터, 서로 다른 운영체제, 서로 다른 문서 포맷 안에 지식이 잠겨 있었고, 사람이 직접 위치를 알아서 찾아가야 했다.

이 불편을 줄이기 위해 [[월드 와이드 웹]]은 "문서를 문서로 연결한다"는 단순한 선택을 한다. 전화번호부처럼 중앙에서 모든 것을 정리하려 한 것이 아니라, 각 문서가 다른 문서를 가리키는 링크를 갖게 했다. 이때 주소 체계([[URL]])와 전송 규칙([[HTTP]])을 함께 둬서, 어디에 있는 문서를 어떤 방식으로 요청할지를 표준화했다.

![[assets/images/웹-클라이언트서버-모델.svg]]

low-level로 보면 동작은 간단하다. 사용자가 브라우저 주소창에 URL을 입력하면 브라우저가 서버로 HTTP 요청을 보낸다. 서버는 HTML 문서를 돌려주고, 브라우저는 그 문서를 해석해 화면에 그린다. 핵심은 "네트워크 위에서 문서를 읽는 방식"을 공통 규약으로 묶었다는 점이다. 이 공통 규약 때문에 운영체제와 하드웨어가 달라도 같은 문서에 접근할 수 있다.

이렇게 하면 어떤 일이 생기냐면, 정보가 파일 단위에서 링크 그래프로 바뀐다. 한 문서를 읽다가 관련 문서로 바로 넘어갈 수 있고, 지식의 탐색 비용이 급격히 줄어든다. 오늘날 검색, 온라인 문서화, 전자상거래, 소셜 플랫폼이 모두 이 구조 위에서 커진 이유가 여기 있다.

물론 초기 웹은 문서 중심이라 상호작용이 약했고, 디자인과 애플리케이션 기능도 제한적이었다. 그래서 이후에 [[CSS]]와 [[JavaScript]]가 등장해 표현과 동작을 분리하고 확장했다. 즉 웹의 출발점은 "하이퍼링크 문서 시스템"이었고, 발전 과정에서 "애플리케이션 플랫폼"으로 진화했다.

## 🔎 확장 질문

★★★★★ 웹 이전의 인터넷 서비스(FTP, Telnet, Gopher)는 무엇이 불편했고, 웹은 무엇을 바꿨는가?

> [!important]
> 기존 서비스는 용도별로 접근 방식이 달라 학습 비용이 높았다. 웹은 URL과 HTTP, HTML을 조합해 "문서 접근 방식"을 통일했다. 사용자는 서비스마다 새 도구를 익히기보다 브라우저 하나로 탐색할 수 있게 됐다.

★★★★☆ URL과 HTTP가 분리된 설계는 왜 중요한가?

> [!important]
> URL은 자원의 위치/식별을 담당하고 HTTP는 전송 규칙을 담당한다. 식별과 전송을 분리하면, 같은 자원 모델 위에서 전송 방식 확장(버전 업, 캐시 정책, 보안 계층 결합)이 쉬워진다.

★★★☆☆ 웹이 문서 시스템에서 애플리케이션 플랫폼으로 바뀐 전환점은 무엇인가?

> [!important]
> 정적 HTML 중심 구조에 CSS와 JavaScript가 결합되면서 화면 표현과 상호작용이 크게 확장됐다. 이후 비동기 통신과 브라우저 엔진 고도화가 이어지며 웹은 문서 열람 도구를 넘어 실행 플랫폼이 됐다.

## 🧠 이해 점검 퀴즈

**Q1 (단답형)** 웹의 핵심 3요소(URL, HTTP, HTML) 중 "전송 규칙"을 담당하는 것은 무엇인가?

> [!important]
> HTTP.

**Q2 (서술형)** 웹이 등장했을 때 정보 접근 비용을 줄인 구조적 이유를 설명하라.

> [!important]
> 링크 기반 문서 구조와 표준 프로토콜 덕분이다. 문서들이 URL로 식별되고 HTTP로 전달되면서, 이기종 시스템 간 문서 접근이 동일한 절차로 가능해졌다. 결과적으로 사용자는 시스템별 접근법을 따로 배우지 않아도 됐다.

**Q3 (설계 의도)** 웹은 왜 중앙집중형 데이터베이스가 아니라 분산 링크 구조를 선택했는가?

> [!important]
> 인터넷 규모에서 중앙 통제형 카탈로그는 확장성과 갱신 비용이 높다. 분산 링크 구조는 각 주체가 문서를 독립적으로 발행하면서도 상호 연결을 유지할 수 있어 확장과 자율성 측면에서 유리하다.
