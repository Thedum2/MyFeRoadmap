# 웹은 무엇이고 왜 탄생하였는가?

#질문

인터넷이 있다고 해서 곧바로 오늘날의 웹이 생기는 것은 아니다. 인터넷은 컴퓨터들이 서로 데이터를 주고받는 기반 네트워크에 가깝고, 웹은 그 위에서 사람이 문서와 정보를 쉽게 연결하고 탐색하도록 만든 서비스 계층이다. 즉 인터넷이 도로망이라면 웹은 그 도로 위를 달리는 공공 정보 체계에 가깝다.

웹이 탄생한 배경에는 정보의 분산 문제가 있었다. 연구자와 조직은 서로 다른 컴퓨터에 문서를 보관하고 있었고, 그 문서를 일관된 방식으로 찾고 연결하기 어려웠다. "문서마다 주소를 붙이고, 공통 형식으로 표현하고, 클릭 한 번으로 이어 가자"라는 아이디어가 [[월드 와이드 웹]]으로 정리됐다.

![[assets/images/웹-클라이언트서버-모델.svg]]

웹의 핵심 선택은 단순하지만 강력했다. 문서를 식별하는 [[URL]], 문서를 주고받는 [[HTTP]], 문서를 구조화하는 [[HTML]]을 조합한 것이다. 비유하면 책마다 고유한 서가 번호를 붙이고, 모든 도서관이 같은 대출 규칙을 따르며, 책 내부도 공통 편집 규칙으로 쓰게 만든 셈이다.

이 구조 덕분에 브라우저는 특정 서버의 특정 문서를 가져와 공통 방식으로 해석할 수 있었다. 나중에 [[CSS]]와 [[JavaScript]]가 추가되면서 웹은 정적인 문서 모음에서 표현과 상호작용을 가진 플랫폼으로 진화했다. 하지만 출발점은 여전히 문서 연결과 접근성에 있다.

웹이 중요해진 이유는 폐쇄적인 애플리케이션 배포 방식과 달리, 열린 표준 위에서 누구나 문서를 발행하고 접근할 수 있었기 때문이다. 하나의 운영체제나 특정 회사의 런타임에 강하게 묶이지 않고, 브라우저라는 공통 클라이언트만 있으면 접속할 수 있다는 점이 폭발적인 확산을 만들었다.

결국 웹은 인터넷의 한 응용이면서도, 인간이 정보를 조직하고 연결하는 방식을 크게 바꾼 계층이다. 웹은 단순한 사이트 모음이 아니라, 주소 지정과 전송, 문서 구조라는 약속을 통해 정보 생태계를 공통 언어로 만든 시스템이다.

---

## 프론트엔드 개발자로써 이 내용을 활용할때 주의할 점

웹을 앱 껍데기로만 보면 본질을 놓친다. 웹은 문서, 링크, 접근성, 개방성을 기반으로 성장한 플랫폼이므로 그 특성을 거스르는 설계는 장기적으로 사용자 경험을 해친다.

실제 활용 단계에서는 URL 설계, 링크 가능성, 시맨틱 구조, 점진적 향상을 항상 의식해야 한다. 화면을 화려하게 만드는 것보다, 웹답게 탐색 가능하고 공유 가능하며 접근 가능한 구조를 유지하는 것이 더 중요할 때가 많다.

## 🔎 확장 질문

★★★★★ 인터넷과 웹을 구분해서 이해해야 하는 이유는 무엇인가?

> [!important]
> 인터넷은 전송 기반이고, 웹은 그 위에서 동작하는 정보 시스템이다. 둘을 구분해야 네트워크 문제와 웹 구조 문제를 정확히 분리해 이해할 수 있다.

★★★★☆ 웹 표준이 개방형 생태계를 유지하는 데 왜 중요한가?

> [!important]
> 공통 규약이 있어야 특정 벤더에 종속되지 않고 문서와 서비스가 상호 운용될 수 있다. 웹의 확장성은 개방형 표준 위에서 나왔다.

★★★☆☆ 모바일 앱 시대에도 웹이 여전히 강한 이유는 무엇인가?

> [!important]
> 링크 공유, 즉시 접근, 배포 장벽의 낮음, 검색 가능성 같은 웹 고유의 장점이 여전히 유효하기 때문이다.

**Q1 (단답형)** 문서를 식별하는 웹의 핵심 주소 체계는 무엇인가?

> [!important]
> URL.

**Q2 (서술형)** 웹이 왜 인터넷과 다른 개념인지 설명하라.

> [!important]
> 인터넷은 데이터를 전달하는 네트워크 기반이고, 웹은 그 위에서 URL, HTTP, HTML 같은 공통 규약으로 문서를 연결하고 제공하는 응용 계층이다.

**Q3 (설계 의도)** 웹은 왜 문서 주소, 전송 규약, 문서 형식을 분리한 구조로 설계되었는가?

> [!important]
> 서로 다른 시스템이 같은 방식으로 문서를 찾고 전달하고 해석하게 만들기 위해서다. 역할을 분리해야 개방성과 상호 운용성이 높아진다.
