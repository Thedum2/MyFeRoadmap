# 인터넷은 어떻게 동작하는가?
#질문

## 1. 등장 배경
초기의 네트워크는 "연결" 자체가 목표였다. 연구소와 군, 대학이 서로 다른 장비를 붙여 데이터만 주고받을 수 있으면 성공이었다. 문제는 네트워크가 커질수록 한 가지 방식으로 모든 컴퓨터를 직접 연결하는 구조가 버티지 못했다는 점이다. 회선 하나가 끊기면 통신 전체가 흔들렸고, 중앙 장비에 장애가 나면 넓은 구간이 같이 멈췄다.

그래서 인터넷은 "항상 같은 길로 보내는 회선 중심"이 아니라 "작게 쪼개서 여러 길로 보내는 방식"을 택했다. 이 전환이 [[패킷 스위칭]]이다. 데이터는 작은 패킷으로 나뉘고, 네트워크는 각 패킷을 그때그때 가능한 경로로 전달한다. 목표는 완벽한 단일 경로가 아니라, 일부 장애가 있어도 전체가 계속 동작하는 생존성이다.

## 2. 구조적 전환
인터넷은 설계를 계층으로 분리했다. 장비 제조사나 운영체제가 달라도 통신이 되게 하려면, 무엇을 어디서 책임지는지 경계를 명확히 해야 했기 때문이다. 이때 핵심 기준이 [[TCP IP]] 스택이다.

하위 계층은 비트를 실제 매체로 흘려보내고, 중간 계층은 패킷을 주소 기반으로 전달하고, 상위 계층은 애플리케이션이 이해할 데이터 흐름을 맞춘다. 이름 해석은 [[DNS]], 하이퍼텍스트 요청/응답은 [[HTTP]], 실제 브라우저-서버 상호작용은 [[클라이언트 서버 모델]]이 맡는다. 즉 인터넷은 "단일 거대 프로토콜"이 아니라, 느슨하게 결합된 프로토콜 집합으로 확장성을 확보했다.

## 3. 내부 동작 원리 (Low-Level)
브라우저에서 주소를 입력하면 먼저 도메인을 IP로 바꾼다. 로컬 캐시, OS 캐시, 리졸버를 차례로 거쳐 [[DNS]] 질의가 발생한다. IP를 얻으면 송신 측은 데이터를 세그먼트로 나누고, 각 조각에 포트 번호, 시퀀스 번호, 체크섬 같은 제어 정보를 붙인다.

그다음 네트워크 계층에서 IP 헤더가 붙는다. 목적지 IP, TTL, 조각화 관련 정보가 들어간다. 이 패킷은 라우터를 지날 때마다 TTL이 줄고, 라우터는 자신의 포워딩 테이블을 보고 다음 홉을 고른다. 이 과정 전체가 [[라우팅]]이다.

라우터의 테이블은 내부망에서는 IGP, 자율 시스템 간에는 [[BGP]] 같은 정책 기반 경로 교환으로 갱신된다. 그래서 "가장 짧은 경로"가 아니라 "정책상 허용된 경로"가 선택되기도 한다. 인터넷이 기술만이 아니라 운영 주체 간 계약의 집합인 이유다.

전송 계층에서는 손실 복구와 속도 조절이 핵심이다. TCP는 ACK 기반 재전송과 윈도우 기반 흐름 제어를 사용한다. 네트워크가 혼잡해지면 송신 속도를 줄이는 [[혼잡 제어]]가 작동한다. 결국 인터넷의 신뢰성은 "절대 손실 없음"이 아니라 "손실을 전제로 복구"하는 방식으로 얻어진다.

> [!important]
> 인터넷의 low-level 핵심은 캡슐화와 디캡슐화다. 보내는 쪽은 계층마다 헤더를 붙이고, 받는 쪽은 역순으로 벗겨 의미를 복원한다.

## 4. 활용
메신저 한 줄 전송도 내부에서는 여러 단계가 겹친다. 앱은 HTTPS 요청을 만들고, 운영체제는 소켓 버퍼에 적재하고, NIC는 프레임으로 내보낸다. 중간 라우터들은 패킷을 릴레이하고, 수신 서버는 커널 네트워크 스택을 거쳐 사용자 프로세스에 데이터를 전달한다.

동영상 스트리밍은 같은 원리를 더 크게 사용한다. CDN 엣지 서버를 가까운 위치에 두고, TCP/QUIC 특성에 맞춰 세그먼트를 조절하며, 손실 상황에서는 비트레이트를 낮춰 재생 끊김을 줄인다. 사용자는 "재생 버튼"만 누르지만, 뒤에서는 주소 해석, 경로 선택, 재전송, 혼잡 대응이 동시에 일어난다.

## 5. 한계와 이후 발전
인터넷은 개방성과 상호운용성 덕분에 폭발적으로 성장했지만, 그 대가도 분명했다. 주소 고갈은 IPv6 전환을 밀어냈고, 경로는 기술적으로 가능해도 정책상 우회되는 경우가 많다. 지연에 민감한 서비스에서는 TCP의 헤드 오브 라인 블로킹이 체감 품질을 떨어뜨리기도 했다.

그래서 이후 발전은 "기본 원칙은 유지하고 병목만 교체"하는 방향이었다. HTTP 2/3, QUIC, Anycast, 대형 CDN, 제로 트러스트 기반 보안 모델은 인터넷 전체를 뒤엎기보다, 기존 계층 위에서 성능과 안정성을 재설계했다.

## 6. 🔎 확장 질문
- ★★★★★ 왜 인터넷은 회선 교환이 아니라 패킷 스위칭을 채택했는가?
> [!important]
> 회선 교환은 연결이 고정되어 장애와 자원 낭비에 취약하다. 패킷 스위칭은 경로를 동적으로 선택해 부분 장애 상황에서도 전체 네트워크 생존성을 높인다.

- ★★★★☆ BGP는 기술적으로 최단 경로보다 정책 경로를 우선하는가?
> [!important]
> BGP는 자율 시스템 운영 정책을 반영하는 경로 벡터 프로토콜이다. 비용, 피어링 계약, 보안 정책이 경로 선택에 반영되므로 최단 거리와 다른 결과가 흔하다.

- ★★★☆☆ DNS 캐시 계층은 왜 여러 단계로 분리되어 있는가?
> [!important]
> 지연 감소와 질의 부하 분산 때문이다. 브라우저/OS/리졸버/권한 서버 캐시가 계층적으로 동작해 평균 응답 시간을 줄이고 상위 권한 서버의 부하를 완화한다.

## 7. 🧠 이해 점검 퀴즈
- 단답형: 패킷이 라우터를 지날 때 무한 순환을 방지하기 위해 줄어드는 IP 헤더 필드는 무엇인가?
> [!important]
> TTL(Time To Live)

- 서술형: 사용자가 `https://example.com`을 입력한 뒤 첫 바이트를 받기까지의 과정을 DNS, TCP, TLS, HTTP 순서로 설명하라.
> [!important]
> 도메인 이름을 DNS로 IP로 해석한 뒤 TCP 연결을 수립한다. 이어 TLS 핸드셰이크로 암호화 세션을 합의하고, HTTP 요청을 전송한다. 서버는 HTTP 응답의 첫 바이트를 반환하고, 클라이언트는 이를 읽어 렌더링 파이프라인을 시작한다.

- 왜 이런 설계를 했는가?: 인터넷은 왜 중앙 통제형 단일 프로토콜 대신 계층형 프로토콜 집합으로 설계되었는가?
> [!important]
> 전 세계 이기종 장비와 조직이 동시에 참여하려면 구현 독립성과 점진적 업그레이드가 필요했기 때문이다. 계층 분리는 하위 매체가 바뀌어도 상위 서비스가 유지되게 해 장기 확장성과 상호운용성을 보장한다.
