# 온프레미스 클라우드 방식의 환경 차이는?

#질문

회사가 서비스를 처음 만들던 시절에는 서버를 직접 사서 전산실에 놓는 게 기본이었다. 트래픽이 늘면 장비를 추가 구매하고, 전원과 냉각, 네트워크, 백업 장치까지 스스로 책임졌다. 이게 [[온프레미스]]의 출발점이다. 내 건물을 내가 관리하는 방식이라 통제권은 강하지만, 준비 시간과 초기 비용이 크다.

반대로 [[퍼블릭 클라우드]]는 이미 지어진 거대한 데이터센터의 일부를 임대하는 모델이다. 서버를 "구매"하지 않고 필요할 때 "할당"한다. 새 서비스 실험이 필요하면 장비 발주 대신 콘솔이나 API 호출로 자원을 몇 분 안에 만들 수 있다. 그래서 팀이 처음 고민하는 질문이 생긴다. 통제권을 더 가질지, 속도와 탄력성을 더 가질지다.

![[assets/images/클라우드-컴퓨팅-구조.svg]]

low-level에서 차이는 리소스 프로비저닝 경로에서 가장 크게 드러난다. 온프레미스에서는 물리 서버 위에 [[가상화]] 계층을 직접 운영하고, 장애 디스크 교체나 펌웨어 업데이트까지 운영 조직의 일이다. 클라우드에서는 같은 가상머신이라도 그 아래 물리 인프라와 기본 네트워크 운영을 사업자가 숨겨준다. 사용자는 인스턴스, 스토리지, 로드밸런서를 API로 조립해 환경을 만든다.

이렇게 보면 둘 다 결국 컴퓨팅 자원을 쓰는 방식인데, 책임 경계가 다르다. 이 경계를 설명할 때 [[Shared Responsibility Model]]을 쓴다. 온프레미스는 하드웨어부터 애플리케이션까지 대부분을 직접 책임지고, 클라우드는 서비스 모델(IaaS/PaaS/SaaS)에 따라 책임이 분리된다. 클라우드가 "운영이 필요 없다"가 아니라, 운영의 무게중심이 인프라 유지보수에서 보안 설정, 권한 관리, 비용 통제로 이동한 것이다.

비용 구조도 설계 결정에 직접 영향을 준다. 온프레미스는 주로 CAPEX(선투자) 성격이라 예측 가능하지만 초기 진입 장벽이 높다. 클라우드는 OPEX(사용량 기반) 성격이라 시작은 가볍지만, 아키텍처가 비효율적이면 비용이 빠르게 누적된다. 그래서 클라우드 팀은 성능 튜닝만큼이나 리소스 크기, 자동 확장 정책, 예약 할인 같은 비용 아키텍처를 같이 설계한다.

운영 자동화에서도 차이가 난다. 온프레미스든 클라우드든 사람이 클릭으로 환경을 만들면 재현성이 떨어진다. 그래서 [[Infrastructure as Code]]가 중요해진다. 차이는 자동화의 효과가 나타나는 속도다. 클라우드는 API 기반 리소스가 풍부해서 IaC를 적용하면 같은 환경을 빠르게 복제하고 폐기할 수 있다. 개발, 스테이징, 운영을 일관되게 맞추기 쉬워진다.

실무에서는 하나만 고집하지 않는 경우가 많다. 규제가 강한 데이터는 온프레미스에 두고, 탄력 확장이 필요한 웹 계층은 클라우드에 두는 하이브리드 구성이 대표적이다. 결국 선택 기준은 유행이 아니라 데이터 민감도, 배포 속도, 운영 인력 역량, 비용 예측 가능성이다.

```mermaid
flowchart TB
    subgraph OnPrem[온프레미스]
      HW[물리 서버/스토리지/네트워크]
      HV[[하이퍼바이저]]
      VM[VM/컨테이너]
      APP[애플리케이션]
      HW --> HV --> VM --> APP
    end

    subgraph Cloud[퍼블릭 클라우드]
      CP[클라우드 API/콘솔]
      IaaS[IaaS 리소스]
      APP2[애플리케이션]
      CP --> IaaS --> APP2
    end

    SR[[Shared Responsibility Model]] --- OnPrem
    SR --- Cloud
```

## 🔎 확장 질문

★★★★★ 하이브리드 클라우드에서 네트워크 경계와 보안 경계를 어떻게 설계해야 하는가?

> [!important]
> 하이브리드는 연결성이 높아질수록 공격면도 넓어진다. 전용선/VPN, 세그먼트 분리, 제로트러스트 접근제어, 키 관리 분리를 함께 설계해야 한다. 핵심은 "연결"보다 "경계의 명시"다. 어떤 트래픽이 어디를 통과하는지와 누가 어떤 자원에 접근하는지를 먼저 모델링해야 사고를 줄일 수 있다.

★★★★☆ 클라우드 전환 시 리프트 앤 시프트와 리팩터링 중 무엇을 먼저 선택해야 하는가?

> [!important]
> 시간 압박이 크면 리프트 앤 시프트로 이전 리스크를 낮추고, 이후 병목 구간부터 점진 리팩터링하는 전략이 현실적이다. 반대로 트래픽 패턴이 급격히 변하거나 비용 최적화가 핵심이면 초기부터 클라우드 네이티브 구조로 일부 재설계하는 편이 장기 비용을 줄인다.

★★★☆☆ 온프레미스가 여전히 유리한 워크로드는 어떤 특성을 가지는가?

> [!important]
> 규제상 데이터 반출이 어렵거나, 장기간 안정적 고정 부하를 처리하며, 하드웨어 특수 장비 제어가 필요한 워크로드는 온프레미스가 유리할 수 있다. 특히 예측 가능한 고정 사용량에서는 장비 감가상각 모델이 클라우드 사용량 과금보다 경제적일 수 있다.

## 🧠 이해 점검 퀴즈

**Q1 (단답형)** 온프레미스와 퍼블릭 클라우드의 가장 큰 구조적 차이를 한 문장으로 쓰면?

> [!important]
> 인프라 소유와 운영 책임의 경계가 다르다.

**Q2 (서술형)** 클라우드가 초기 도입은 빠르지만 장기 운영에서 비용 이슈가 생기는 이유를 설명하라.

> [!important]
> 클라우드는 사용량 기반 과금이라 초기 투자 없이 빠르게 시작할 수 있다. 하지만 과도한 리소스 할당, 잘못된 스토리지/네트워크 설계, 미사용 자원 방치가 누적되면 비용이 계속 발생한다. 즉, 기술 아키텍처와 비용 아키텍처를 함께 최적화하지 않으면 장기적으로 비싸질 수 있다.

**Q3 (설계 의도)** 왜 많은 조직이 단일 방식 대신 하이브리드 환경을 선택하는가?

> [!important]
> 시스템마다 요구사항이 다르기 때문이다. 민감 데이터 통제와 규제 준수는 온프레미스가 유리하고, 탄력 확장과 빠른 실험은 클라우드가 유리하다. 하이브리드는 서로 다른 요구를 분리 배치해 전체 리스크와 속도를 동시에 관리하려는 설계 선택이다.
