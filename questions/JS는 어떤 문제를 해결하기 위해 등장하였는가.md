# JS는 어떤 문제를 해결하기 위해 등장하였는가?

#질문

초기 웹은 "읽는 문서"에는 강했지만 "반응하는 화면"에는 약했다. 사용자가 버튼을 누르거나 입력을 바꿔도 즉시 처리하기 어렵고, 작은 상호작용도 서버에 요청해 새 페이지를 받아야 했다. 이는 지연과 깜빡임, 과도한 네트워크 비용으로 이어졌다.

[[JavaScript]]는 이 문제를 해결하기 위해 등장한다. 핵심은 브라우저 안에서 코드를 실행해 사용자 이벤트를 즉시 처리하는 것이다. 비유하면, 매번 본사 승인(서버 왕복)을 기다리던 매장을, 현장에서 바로 판단 가능한 지점으로 바꾼 셈이다.

![[assets/images/JavaScript-이벤트루프.png]]

low-level에서는 브라우저 엔진이 스크립트를 실행하고, 이벤트 발생 시 콜백을 큐에 쌓아 이벤트 루프로 처리한다. DOM 조작, 입력 검증, 비동기 요청 처리 같은 작업이 페이지 컨텍스트에서 이루어지므로 사용자 반응성이 크게 올라간다. 서버는 데이터와 권한 검증 중심으로, 클라이언트는 상호작용 중심으로 역할이 분화된다.

이렇게 하면 어떤 일이 생기냐면, 페이지 전체를 다시 불러오지 않아도 필요한 부분만 갱신할 수 있다. 폼 검증, 자동완성, 무한 스크롤, 실시간 알림처럼 현대 웹앱의 기본 UX가 가능해진다.

한계도 생긴다. 클라이언트 코드가 커지면 초기 로딩 비용과 복잡도가 올라가고, 보안 취약점(XSS 등) 관리가 중요해진다. 그래서 번들 분할, CSP, 타입 시스템, 프레임워크 아키텍처가 함께 발전했다. 결국 JavaScript는 웹을 정적 문서에서 상호작용 플랫폼으로 전환한 핵심 동력이었다.

## 🔎 확장 질문

★★★★★ JavaScript의 비동기 모델(이벤트 루프)은 왜 UI 반응성에 중요한가?

> [!important]
> 긴 작업이 메인 실행 흐름을 독점하면 입력 지연이 커진다. 이벤트 루프 기반 비동기 처리는 작업을 분할/예약해 사용자 입력 처리 기회를 확보하므로 체감 반응성을 유지한다.

★★★★☆ 서버 사이드 검증이 있는데도 클라이언트 검증이 필요한 이유는 무엇인가?

> [!important]
> 서버 검증은 보안과 무결성 보장을 위해 필수고, 클라이언트 검증은 즉각적 피드백으로 UX를 개선한다. 둘은 대체 관계가 아니라 목적이 다른 보완 관계다.

★★★☆☆ JavaScript 도입이 프론트엔드 아키텍처 복잡도를 높인 이유는 무엇인가?

> [!important]
> 상태 관리, 비동기 흐름, 컴포넌트 수명주기, 번들 최적화 같은 문제가 새롭게 생기기 때문이다. 기능 확장성과 복잡도는 트레이드오프 관계에 있다.

## 🧠 이해 점검 퀴즈

**Q1 (단답형)** JavaScript의 비동기 작업 스케줄링과 관련해 자주 언급되는 실행 모델은?

> [!important]
> 이벤트 루프(Event Loop).

**Q2 (서술형)** JavaScript가 없던 웹과 있던 웹의 사용자 상호작용 차이를 설명하라.

> [!important]
> JavaScript가 없으면 사용자 상호작용 처리를 위해 서버 왕복과 페이지 재로딩에 의존한다. JavaScript가 있으면 브라우저 내에서 이벤트를 즉시 처리하고 필요한 부분만 갱신할 수 있어 반응성이 높아진다.

**Q3 (설계 의도)** 웹에 스크립트 실행 계층을 추가한 이유는 무엇인가?

> [!important]
> 정적 문서 전달만으로는 복잡한 사용자 인터랙션 요구를 만족할 수 없기 때문이다. 실행 계층을 추가해 클라이언트에서 즉시 계산/검증/갱신을 수행하도록 한 것이 설계 핵심이다.
