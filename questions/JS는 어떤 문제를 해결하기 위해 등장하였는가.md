# JS는 어떤 문제를 해결하기 위해 등장하였는가?

#질문

초기의 웹페이지는 잘 정리된 전단지에 가까웠다. 문서를 내려받아 읽고, 링크를 눌러 다른 문서로 이동하는 흐름이 중심이었다. 그런데 사용자는 금방 더 많은 것을 원했다. 버튼을 누르면 즉시 반응하고, 입력값을 서버에 보내기 전에 검사하고, 화면 일부만 바꾸면서 상호작용하는 경험 말이다.

문제는 [[HTML]]과 [[CSS]]만으로는 이 요구를 처리하기 어렵다는 점이었다. HTML은 구조를, CSS는 표현을 다루지만, "사용자의 행동에 따라 지금 이 순간 어떤 로직을 실행할 것인가"는 별도 계층이 필요했다. 그래서 등장한 것이 [[JavaScript]]다.

비유하면 HTML이 무대의 세트이고 CSS가 조명과 의상이라면, JavaScript는 배우의 동선과 타이밍을 지휘하는 무대 감독에 가깝다. 버튼 클릭, 입력 이벤트, 비동기 응답, 타이머 같은 시간성 있는 일을 처리해야 웹이 문서에서 애플리케이션으로 넘어갈 수 있었다.

![[assets/images/JavaScript-이벤트루프.png]]

브라우저 내부에서 JavaScript는 [[DOM]]을 읽고 수정할 수 있고, 네트워크 요청을 보내고, 이벤트를 등록할 수 있다. 이 과정은 [[이벤트 루프]]를 중심으로 움직인다. 콜스택에서 동기 코드를 실행하고, 완료된 비동기 작업은 큐에 쌓아 두었다가 순서대로 처리한다. 덕분에 브라우저는 한 스레드 중심 구조에서도 사용자 입력과 네트워크 응답을 끊김 없이 조율할 수 있다.

이렇게 하면 어떤 일이 생기냐면, 웹페이지는 더 이상 완성된 문서를 한 번 내려받아 끝내는 구조가 아니라, 사용자 행동에 맞춰 상태를 바꾸는 실행 환경이 된다. 장바구니 수량 버튼, 자동완성 검색창, 무한 스크롤, 실시간 알림은 모두 이 계층 위에서 가능해졌다.

물론 JavaScript 역시 한계를 만들었다. 동적 로직이 강해질수록 코드 복잡도가 커지고, DOM 조작이 잦아지면 성능 문제도 발생한다. 그래서 프레임워크, 가상 DOM, 상태 관리, 빌드 도구가 함께 발전했다. 하지만 출발점은 단순하다. JavaScript는 "웹페이지를 살아 있는 인터페이스로 만들기 위해" 등장했다.

짧게 정리하면, JavaScript는 예쁜 화면을 만드는 언어가 아니라 시간과 상호작용을 처리하는 언어다. 웹이 정적인 문서에서 동적인 애플리케이션으로 진화한 지점에는 항상 JavaScript가 있었다.

---

## 프론트엔드 개발자로써 이 내용을 활용할때 주의할 점

JavaScript를 많이 쓴다고 좋은 UI가 되는 것은 아니다. 사용자 상호작용에 필요한 최소한의 로직과, 브라우저 기본 기능으로 해결할 수 있는 부분을 구분해야 한다.

실제 활용 단계에서는 DOM 조작 빈도, 이벤트 등록 위치, 비동기 흐름 관리가 중요하다. 특히 입력 지연과 메인 스레드 블로킹은 결국 JavaScript가 UI 시간을 얼마나 점유하느냐의 문제로 돌아온다.

## 🔎 확장 질문

★★★★★ JavaScript의 이벤트 루프는 왜 UI 반응성과 직결되는가?

> [!important]
> 메인 스레드에서 긴 작업이 실행되면 입력 처리와 렌더링이 뒤로 밀린다. 이벤트 루프 구조를 이해해야 끊김 없는 UI를 설계할 수 있다.

★★★★☆ 프레임워크는 JavaScript의 어떤 복잡도를 줄이기 위해 등장했는가?

> [!important]
> 상태 변화에 따른 DOM 업데이트를 사람이 직접 관리하면 코드가 금방 얽힌다. 프레임워크는 그 변경 과정을 추상화해 예측 가능성을 높인다.

★★★☆☆ JavaScript가 없으면 현대 웹은 어디까지 가능할까?

> [!important]
> 문서 제공과 기본 폼 제출은 가능하지만, 풍부한 상호작용과 부분 갱신, 실시간 상태 반영은 크게 제한된다. 웹은 다시 정적인 문서 중심으로 돌아간다.

## 🧠 이해 점검 퀴즈

**Q1 (단답형)** 브라우저에서 비동기 작업 완료 후 콜백을 처리하는 흐름의 중심 메커니즘은 무엇인가?

> [!important]
> 이벤트 루프.

**Q2 (서술형)** JavaScript가 HTML과 CSS만으로 해결할 수 없던 문제를 설명하라.

> [!important]
> 사용자 입력, 시간 기반 작업, 네트워크 응답에 따라 화면과 상태를 동적으로 바꾸는 로직을 처리해야 했는데, HTML과 CSS는 구조와 표현에만 집중하므로 이런 실행 흐름을 담당할 수 없었다.

**Q3 (설계 의도)** 웹은 왜 별도 스크립트 언어를 도입했는가?

> [!important]
> 문서 구조와 표현만으로는 상호작용을 만들 수 없었기 때문이다. 시간과 이벤트, 상태 변화를 처리하는 계산 계층이 필요했다.
